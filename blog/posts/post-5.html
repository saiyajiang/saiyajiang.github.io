Git 工作流最佳实践：团队协作的那些坑

2026年4月5日 约 7 分钟阅读
#Git#工程实践

Branch 策略、commit message 规范、PR review 流程——总结了一些在实际协作中踩过的坑和解决方案。

Branch 策略

推荐 Gitflow 简化版：

  • main — 稳定分支，永远可以直接部署
  • develop — 开发分支，集成所有功能
  • feature/xxx — 新功能分支
  • hotfix/xxx — 紧急修复分支

Commit Message 规范

好的 commit message 是不需要看代码就能理解改了什么。
feat: 添加用户注册功能
fix: 修复登录页面白屏问题
docs: 更新 README
refactor: 重构支付模块
test: 添加单元测试

PR 的正确姿势

  • PR 越小越好，一个 PR 只做一件事
  • PR 描述写清楚：改了什么、为什么改、如何测试
  • 自审后再请人 review，别把 code review 当成 debug 服务
  • review 时先问"这个改法会不会影响其他地方"，再提细节

最容易踩的坑

不要在 main 分支上直接改代码。哪怕是小改动，也要走 PR 流程。

不要 force push 到已合并的分支。历史记录是团队的记忆，force push 会让所有人的本地历史和你不一致。
